두 개의 SRS 문서(v0.1 및 v0.2 Opus 버전)를 비교 검토한 결과, **v0.2는 초기 버전의 엔터프라이즈 지향적 설계를 '린(Lean)한 MVP 실행안'으로 구체화하고 단계별 로드맵을 도입**한 것이 핵심적인 차이입니다.

요청하신 세 가지 관점에 따른 상세 비교 보고서입니다.

---

### 1) 기술 스택의 명확성 (Technical Stack Clarity)
v0.2에서는 초기부터 고비용/고난이도 기술을 도입하는 대신, 서비스 성장에 맞춰 기술 스택을 확장하는 방식을 채택했습니다.

| 항목 | v0.1 (Enterprise Ready) | v0.2 (Phase-based MVP) | 주요 변경 사항 |
| :--- | :--- | :--- | :--- |
| **파싱 인프라** | HWP/Word/PDF 통합 관리 (전용 OCR 명시) | PDF/Word 우선 도입(P0), HWP 후순위(P2) | 기술적 리스크(HWP) 분산 및 단계적 구현 |
| **모니터링** | DataDog APM, PagerDuty | CloudWatch, Slack Webhook (P1~2) | 초기 운영 비용 절감 및 오버엔지니어링 방지 |
| **서버/DB** | ECS Fargate, Redis (초기 필수) | t3 인스턴스 → ECS Fargate (P2~) 확장 | 트래픽 규모에 따른 유연한 인프라 운용 |
| **데이터 보안** | AES-256 상시 적용 | SSE-S3(P1) → AES-256(P2) 단계적 강화 | 인프라 기본 기능을 활용하여 초기 속도 확보 |
| **아키텍처** | MSA 또는 모듈형 모놀리스 | **Modular Monolith** (MSA 옵션 제거) | 1인/소규모 개발팀의 운영 효율성 극대화 |

### 2) MVP 목표의 조정 내용 (MVP Goal Adjustments)
v0.2에서는 '모든 기능을 한 번에' 만드는 방식에서 '가설을 검증하며 확장'하는 방식으로 전략이 수정되었습니다.

| 항목 | v0.1 목표 | v0.2 조정 내용 | 조정 사유 |
| :--- | :--- | :--- | :--- |
| **개발 로드맵** | 단일 상용화 버전 (v1.0) | **4단계 Roadmap (Phase 0~3)** 도입 | 단계별 Phase Gate(검사점)를 통한 리스크 관리 |
| **HWP 지원** | 즉시 지원 (비용 20원 상한) | Phase 2 이관 (단가 100원 현실화) | HWP 파싱 난이도 및 상용 API 비용 고려 |
| **성능/가용성** | SLA 99.9% (상시) | P0~2: 99.5%, P3: 99.9% | 개발 단계별 현실적인 품질 목표 설정 |
| **트래픽 수용** | 초기 동시 1,000명 기준 | P0~1: 동시 50명 이하 타겟팅 | 초기 서버 부하를 낮추어 불필요한 비용 제거 |
| **비용 관리** | 월 $500 고정 예산 | $50(P0) ~ $1,200(P3) 차등 예산 | 비즈니스 확장성(Scalability)에 따른 예산 연동 |

### 3) 기타 차이점 (Other Differences)
문서의 구조와 관리 측면에서도 실무적인 보완이 이루어졌습니다.

| 항목 | v0.1 | v0.2 | 비고 |
| :--- | :--- | :--- | :--- |
| **문서 이력 및 근거** | 초안 (Rev 1.0) | 적절성 검토 결과(REF-07) 반영 (Rev 0.9) | 변경 사유와 근거 문서를 명확히 명시 |
| **이해관계자 참여** | 모든 이해관계자 상시 참여 | 역할별 **참여 Phase 명시** (예: 유팀장 P2~) | 커뮤니케이션 및 업무 분담의 효율성 증대 |
| **검증 지표** | NFR 수치 만족 중심 | **Phase Gate(검증 기준)** 추가 | 기술적 성능뿐 아니라 '전환율' 등 비즈니스 성과 연동 |
| **사용자 가이드** | 일반적인 업로드 UI | 파싱 한계 안내 모달(F6) 강화 | 사용자 기대치 관리(Expectation Mgmt) 구체화 |

---

**요약 보고:**
v0.2는 **"먼저 PDF/Word로 유료 전환율(5%)을 검증(Phase 1)하고, 성공 시 HWP와 쿼터 기능(Phase 2)을 확장한다"**는 전술적 유연성을 SRS에 성공적으로 반영하였습니다. 이는 초기 개발 리스크를 최소화하면서도 리서치 대행사라는 고도화된 타겟(유팀장)까지 확장할 수 있는 현실적인 로드맵입니다.
