## 1. Executive Summary

현재 SRS_v0.1_Opus는 **상용(Production-grade) 수준의 완결성**을 갖추고 있으며, MVP 단계에서 요구하기에는 과도한 범위의 요구사항을 포함하고 있다. 주요 발견은 다음과 같다:

| 평가 축 | 현재 SRS 상태 | 조정 필요 여부 |
|---|---|---|
| **개발 난이도** | 고급 개발자(7년+) 기준으로도 2~3개월 소요. 중급 이하에서는 사실상 완전 구현 불가 | 🔴 Phase 분리 필수 |
| **개발 속도** | 외부 연동 7개, DB 엔티티 8개, API 11개를 MVP 1회에 모두 구현 시 **4~8개월** 예상 | 🔴 범위 축소 필요 |
| **기술 스택 복잡성** | HWP 파싱(저성숙 생태계), Redis 원자적 쿼터, PG 결제 등 고위험 기술 밀집 | 🟠 대안 기술 검토 필요 |
| **운영 비용** | SRS 명시 예산(월 500 USD) 대비 실제 예상 비용(월 **800~1,400 USD**) 초과 가능 | 🔴 예산 재산정 또는 아키텍처 조정 필요 |

> [!CAUTION]
> 현재 SRS를 그대로 MVP 개발 목표로 채택할 경우, **개발 기간 초과, 예산 초과, 기술 부채 누적**의 3중 리스크가 동시에 발생할 가능성이 높다.

### 2.3 구현 난이도 Top 5 (기술 리스크 순)

| 순위 | 요구사항 | 난이도 근거 |
|---|---|---|
| 1 | **HWP 비정형 파싱 + 손실률 1% 미만** (REQ-FUNC-002, 004) | 한국 고유 문서 형식. 오픈소스 파서(pyhwp, hwpx)의 성숙도가 매우 낮고, 복잡한 서식의 HWP에서 텍스트 기반 설문 문항만 정확히 추출하는 것은 산업계에서도 미해결 과제 |
| 2 | **Redis 교차 쿼터 원자적 연산 + Over-quota 1%** (REQ-FUNC-019, 020) | 동시 1,000명 환경에서 성별×연령×지역 교차 쿼터의 Race Condition 방지는 분산 시스템 전문 지식 필요 |
| 3 | **PG 결제 전체 트랜잭션 플로우** (REQ-FUNC-010~013) | 결제 요청→콜백→검증→URL 발급의 4단계. idempotency, 재시도 로직, 결제 실패 엣지 케이스 처리 필수 |
| 4 | **4종 산출물 결측치 0% 자동 생성** (REQ-FUNC-014) | 데이터맵/코드북/변수가이드의 리서치 업계 도메인 지식 + 완벽한 매핑 로직 필요 |
| 5 | **7개 외부 서비스 동시 연동 및 장애 격리** | DataDog, PagerDuty, Slack, Amplitude, GA4, S3, PG — 각각의 인증·에러 핸들링·재시도가 누적. 통합 테스트 복잡도 기하급수적 증가 |

## 3. Part B — 시스템 및 비용 효율성 관점 검토

### 3.1 (B-1) 개발 속도 — MVP 단계 시간 소요 분석

#### 3.1.1 현재 SRS 기준 모듈별 예상 개발 기간

> 기준: 중급(3~5년차) 풀스택 개발자 1인, ENG+AI 방식, 주 40시간 기준

| 모듈 | 예상 기간 | 시간 소요의 주요 원인 |
|---|---|---|
| F1: AI 문서 파서 | **6~8주** | HWP 파서 PoC + 정밀도 튜닝, OCR 모듈 분기 로직, 비동기 파이프라인 구축 |
| F2: ZIP 컴파일러 + Paywall | **3~4주** | 4종 엑셀 생성 로직, PG 결제 연동, S3 Presigned URL, 결제 콜백 핸들링 |
| F3: 워터마크 | **0.5주** | utm 파라미터 삽입, 배너 렌더링 — 단순 작업 |
| F4: 동적 쿼터 | **3~4주** | Redis 원자적 연산, 교차 쿼터 매트릭스, 동시성 테스트 |
| F5: 패널 라우팅 | **1~2주** | HTTP 302, 포스트백 URL CRUD, 상태별 분기 |
| F6: Rate Limit/캐시 | **1주** | Redis 캐시, 해시 기반 중복 감지 |
| F7: 보안/삭제 스케줄러 | **2~3주** | AES-256 적용, Cron 스케줄러, RBAC, 감사 로그 |
| 모니터링 연동 | **2~3주** | DataDog, PagerDuty, Slack, Amplitude, GA4 각각 연동 |
| 프론트엔드 | **4~5주** | 웹 대시보드 + 모바일 웹 폼 + 결제 UI + 쿼터 설정 UI |
| 인프라/DevOps | **2~3주** | AWS 인프라 셋업, CI/CD, 환경 설정 |
| **합계** | **24~33주 (약 6~8개월)** | |

> [!WARNING]
> **MVP에 6~8개월은 과도하다.** 린 스타트업 방법론 기준 MVP는 **4~12주**가 적정 범위이며, 현재 SRS는 이를 2~4배 초과한다. PMF 검증 전에 개발 리소스가 소진될 위험이 있다.

#### 3.1.2 개발 속도 저해 요인 식별

| 저해 요인 | 관련 요구사항 | 영향도 |
|---|---|---|
| **HWP 파서 PoC 불확실성** | REQ-FUNC-001~004 | 🔴 치명적. PoC 실패 시 전체 프로젝트 재설계 필요 |
| **PG 결제 심사·계약 리드타임** | REQ-FUNC-010~013 | 🟠 높음. 토스페이먼츠 가맹점 심사에 2~4주 소요 |
| **외부 패널사(Cint/Toluna) 계약** | REQ-FUNC-023~025 | 🟠 높음. API 접근 권한 획득에 사업자 계약 필요 |
| **도메인 지식 습득** | REQ-FUNC-008, 014 | 🟡 중간. 데이터맵/코드북/변수가이드 산출물의 리서치 업계 표준을 학습해야 함 |
| **모니터링 도구 7개 동시 셋업** | REQ-NF-024~028 | 🟡 중간. 각 도구별 계정 생성, 인증키 발급, 설정에 시간 소요 |

#### 3.1.3 개발 속도 관점 조정 권고

| # | 현행 SRS | 권고 조정안 | 시간 절감 효과 |
|---|---|---|---|
| S-1 | F1에 HWP/Word/PDF 3종 모두 포함 | MVP에서는 **PDF + Word만** 지원. HWP는 Phase 2로 이관 | **3~4주 절감** |
| S-2 | F4(쿼터) + F5(패널 라우팅) MVP 포함 | **Should → Phase 2로 이관**. MVP에서는 단순 응답 수집만 | **4~6주 절감** |

| 충돌 요구사항 | 충돌 내용 |
|---|---|
| REQ-FUNC-002 (HWP 파싱 손실률 1%) | 오픈소스 HWP 파서로는 1% 미만 달성 불가, 상용 API 필요 |

> [!WARNING]
> **CON-05의 문구를 "MVP 단계에서는 비용 효율적 외부 API 활용을 허용하되, 로드맵 상 오픈소스 전환 계획을 수립한다"로 완화하는 것을 권고한다.** 그렇지 않으면 품질 목표(손실률 1%)와 기술 제약(오픈소스 의무화)이 양립 불가능하다.
